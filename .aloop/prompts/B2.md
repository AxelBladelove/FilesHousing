PERFORMANCE-FIRST DIRECTIVE (non-negotiable):

- Choose the leanest viable approach: standard library before dependency, dependency
  only when it clearly beats hand-rolling on speed AND maintenance.
- Know your hot path: pick data structures and algorithms for the access pattern this
  code will actually have; state the expected complexity in a code comment only when
  a slower-looking choice is deliberate.

- Measure when it matters: if the task touches a hot path, add or run a micro-benchmark

## Context

You are working in a Codex project worktree for FilesHousing after B1 has added the Tauri/Rust backend foundation. The frontend UI is ready with mock data and expects `Disk`/`FsNode`-like data for the city map. Your slice is the real backend scanner and aggregation engine. Do not redesign the UI.

## Requirements

1. Replace the placeholder scanner with a real path scanner suitable for Windows-first local drives.
2. Traverse directories iteratively or with bounded recursion, skip inaccessible entries gracefully, and avoid panicking on permission/path errors.
3. Aggregate file metadata into the existing visual tree shape:
   - stable numeric IDs.
   - file/folder name, kind, category, byte size, modified age in days, count, children.
   - path available for backend actions without exposing unsafe operations by default.
4. Categorize files by extension using a lean static mapping aligned with frontend `Cat` values.
5. Keep the response bounded for UI performance: sort by size, retain meaningful top children, and aggregate tiny leftovers if necessary.
6. Implement drive/root listing in a Windows-first way with a safe fallback for other platforms.
7. Add focused Rust tests for classification and aggregation where practical.
8. Commit your work on your worktree branch with clear messages.
